【课程笔记】IT项目管理

  • Liang Hangfeng
  • 课程笔记
  • March 2, 2024

一、基本概念

1.1 价值

1.1.1 项目的价值

组织通过日常工作来维持基本的运行，通过项目来推进自身的发展和壮大。 项目价值是指项目具有明示和潜在的功能，能满足利益相关方明示和潜在的需要。 项目是实现价值、成就事业的载体，通过项目来实现企业和人生的价值，通过三个层次来体现：管理项目、发现项目、创造项目。

1.1.2 项目管理的价值

项目管理是管理学的子集，具有管理学在通常意义上的价值。项目管理最主要的价值来源是通过完成业务目标来获得的，而要确保业务目标的实现，需要有效的项目管理，实行项目管理可以为企业创造巨大价值。

1.2 项目概念

1.2.1 人类活动的特点

目的性、依存性、知识性

1.2.2 作业与项目

作业：连续不断、周而复始的活动；项目：临时的、一次性的活动

1.2.3 项目定义

利用有限资源、在一定时间内，完成满足一系列特定目标的多项相关工作

1.2.4 项目特征

目标独特性、项目一次性、项目整体性、项目临时性、项目不确定性、资源多变性

1.3 项目管理

现代管理科学是从关键路径法（Critical Path Method，CPM）和计划评审技术（Program Evaluation and Review Technique，PERT）的基础上发展，通过融合后发展起来的工作分解结构（Work Breakdown Structure，WBS）、蒙特卡洛（MonteCarlo Simulation，MCS）统计模拟方法和挣值（Earned Value，EV）分析技术，形成关于项目资金、时间、人力等资源控制的管理科学。

1.3.1 管理的概念

管理是管理者为了有效地实现组织目标、个人发展和社会责任，运用管理智能进行协调的过程。 内涵P9

1.3.2 项目管理定义

美国项目管理协会PMI给出定义：项目管理就是将知识、技能、工具与技术应用于项目活动，以满足项目的需要。项目管理通过合理运用和整合项目管理过程得以实现。 五大过程组：启动、规划、执行、监控、收尾

1.3.3 项目管理特点

五项基本任务：项目计划、项目组织、质量管理、费用控制、进度控制 项目管理必须通过不完全确定的过程，在确定的期限内生产出不完全确定的产品或完成不完全确定的任务。 P11

1.3.4 项目管理知识体系

  • PMI和PMBOK

    PMI：美国项目管理协会

    PMBOK：项目管理知识系统

    4大核心知识领域：项目范围管理、项目时间管理、项目成本管理、项目质量管理

    5大辅助知识领域：项目人力资源管理、项目沟通管理、项目风险管理、采购管理和项目于系人管理

  • IPMA与ICB

    IPMA：国际项目管理协会

    ICB：国际项目管理知识体系

    划分为42个要素：28个核心要素、14个附加要素；个人素质的8大特征、总体印象的10个方面

1.4 IT项目

信息技术IT：包括传感器技术、通信技术、计算机技术、控制技术等

1.4.1 定义

信息化的实现以IT项目为载体，以信息技术为基础。 用IT信息化来综合描述信息技术与信息化，与此对应的项目称之为IT信息化项目；有时将以计算机为主体的各种项目称为IT项目。 利用有限资源、在一定时间内，完成满足一系列特定的IT信息化目标和多项相关工作叫做IT项目。

1.4.2 IT项目特征

目标不确定性、需求不稳定性、费用不可控性、项目时限性、对智力的依赖性、项目评价的主观性、项目创新性

1.4.3 IT项目管理

定义：是指在一定期限内，根据一定的IT信息化项目需求，依托一定的资源，为达成一定的IT信息化目标而进行的一系列活动 特点：P24

1.5 软件与软件项目

软件定义、软件分类、软件特点、软件项目特点、软件项目管理意义与重点

二、环境与过程

2.1 组织环境

2.1.1 概念

人类通过社会活动，按照一定目的、任务和形式加以编制的群体称为组织，在存在于组织外部，和组织密切联系，决定组织存在和发展的自然、经济、技术、政治、社会的各种因素和条件的集合称为组织环境。包括自然环境和社会环境两大部分。

2.1.2 特征

客观性、系统性、动态性

2.1.3 战略计划

战略针对的是整体性、长期性、基本性问题。

组织的整体性、长期性、基本性问题为战略计划

战略计划分析方法：SWOT（态势分析）【优势、缺点、机会因素、威胁因素】

2.2 掌握系统方法

2.2.1 系统定义与特征

定义

系统是由相互联系和相互制约的若干组成部分结合而成的、具有特定功能的有机整体。

三层含义：若干部分组成、一定的结构、系统有一定的功能和目的。

特征

集合性（基本特征，一个系统至少由两个及以上的子系统构成，称为要素）、有层次的、相关性（要素之间相互依存与制约）

2.2.2 系统的原理

整体性原理、动态性原理、开放性原理、环境的适应性原理、综合性原理

2.2.3 系统方法

把研究和处理的对象看成一个完整的系统，看到组成部分与各个要素，以及中间的关联关系。

2.3 熟悉项目阶段

2.3.1 项目阶段的特征

IT项目的非独立性、不确定性、隐蔽性等特征，使得项目的范围、工期和团队规模成为支持项目成功的三大支柱。

项目阶段的划分

定义阶段：制定项目建议书，描述为什么做？做什么

开发阶段：规划项目怎么做？谁来做？根据项目建议书来制定更详细的项目计划

实施阶段：执行项目计划，进行项目的监督和控制，根据项目的执行状况形成项目状态报告书

收尾阶段：完成项目的验收和工作总结，形成项目验收报告与工作总结报告

是一个从无到有的过程

一个阶段的结束：通过工作成果的评审标志（判断是否可以进入下一个阶段，最小代价纠错）

2.3.2 项目生命周期特征
  1. 按顺序首尾衔接，通过一些文档、部件来衔接；
  2. 开始对人力费用投入小，中间阶段会增多，最后收尾时也会减少；
  3. 项目开始时完成项目的把握较低（风险和不确定性是最高的），随着项目发展成功概率提高；
  4. 起始阶段项目干系人对项目产品的影响是最大的，随着发展不断下降（随着项目的推进，变更的难度和代价越来越大）
2.3.3 项目生命周期和产品生命周期

产品生命周期从提出研发开始，到停止使用为止。

软件生命周期分为：软件定义、软件开发、软件运行维护 共3个时期。

一个软件的生命周期会通过多个项目来完成。

2.4 了解项目组织

强调项目经历的作用，团队的协作精神。组织形式具有强大的灵活性。

2.5 控制项目过程

过程：一组为实现某个特定目标而进行的一系列相互联系的行动或活动。

两类过程：产品导向型过程、项目管理过程

2.5.1 项目管理过程组

过程组：启动、规划、执行、监督、结束（与四个基本阶段区分）

主要任务

启动：确定并核准项目或项目阶段，形成一个项目章程和一位项目经理

规划：任务的确定和细化，完成工作任务分解结构、项目进度计划和项目预算

执行：实施项目计划，交付实际的项目工作

监控：定期测量和实时监控项目的发展情况

收尾：对项目成果、项目产品、项目阶段进行验收

三、整合项目资源

3.1 概述

项目整合管理：确保项目各要素相互协调所需要的过程。核心为项目资源整合。

项目资源整合：明确项目有哪些资源，并通过整合这些资源以达成项目干系人的愿景。

明确：

  1. 整合项目资源在项目管理种的地位和作用？
  2. 项目的资源有哪些？
  3. 项目干系人指的是？
3.1.1 整合项目资源的意义

”整合项目资源“用于整合项目的各个要素，引导项目走过四个阶段

通过项目资源的整合，将9大领域的相关要素有机结合。

3.1.2 项目资源

静态观点来看

  1. 人力资源：项目干系人
  2. 财务资源：
  3. 项目实物资源：

动态观点来看

  1. 非实物的生产资料：软件等
  2. 无形的管理约束：制度会议等
  3. 项目的运作过程
3.1.3 项目干系人
  1. 客户
  2. 用户
  3. 项目投资人
  4. 项目经理
  5. 项目组成员
  6. 高层管理人员
  7. 反对项目的人
  8. 施加影响者
3.1.4 IT项目经理

地位作用：是项目实施的最高领导者、组织者、责任者（个人责任制），项目有关各方协调配合的桥梁和纽带，处在上下各方的核心地位

职责：

  1. 确保项目目标的实现，保证使客户满意
  2. 制定项目阶段性目标和项目总体控制计划
  3. 组织精干的项目管理班子
  4. 及时决策
  5. 履行合同义务，监督合同执行，处理合同变更

权力：

  1. 指挥权
  2. 人事权
  3. 财权
  4. 技术决策权
  5. 设备、工具、材料的采购与控制权

素质要求：

  1. 道德
  2. 心理
  3. 客户意识
  4. 专业素养
  5. 大局观与战略意识
  6. 项目管理能力
  7. 信心与意志
  8. 胆大心细

3.2 项目启动

项目资源整合的一项重要活动就是成功地启动项目。

”目标驱动、结果引导“是成功启动项目的最好方法。

几个步骤

  1. 了解IT项目背景（是否有结果定义、是否有人做过、是否有钱有权等）
  2. IT项目启动依据（企业战略目标、项目目的、成果说明书等来从战略计划上做考虑）
  3. IT项目启动会议
  4. 制定项目章程：项目整合管理的第一个过程，项目管理47个过程种的第一个，是正式承认项目存在的重要文件，主要干系人签字。（形式不一，如项目立项书、企业需求说明书、产品说明书等）

四、控制项目范围

4.1 项目范围管理概述

4.1.1 项目范围与项目范围管理

项目范围是指产生项目产品阶段包括的所有工作及产生这些产品经过的所有过程。

范围：

  1. 产品范围：某项产品、服务或服务所具备的特性与功能
  2. 工作范围：为具有规定特性与功能的产品、服务或者成果而必须完成的工作

项目的产品范围是制定项目工作范围的基础，而项目工作范围是为了实现项目的产品范围需要做的工作。

项目产品范围和项目工作范围的范围管理必须很好地结合，确保项目工作可以得到项目的最终可交付成果。

4.1.2 项目范围管理的重要性

定义了项目的工作边界，明确了项目的目标和主要的可交付成果。

项目实际要求的 但没有明确定义的工作不能得到有效执行

工作内容不在项目范围内的却被执行，即范围的盲目扩大，影响项目的预算。

作用：

  1. 提高费用、时间和资源估算的准确性
  2. 确定进度测量和控制的基准
  3. 有助于项目分工
4.1.3 项目范围管理过程
  1. 规划范围管理
  2. 收集需求
  3. 定义范围
  4. 创建WBS
  5. 确认范围
  6. 控制范围

4.2 项目范围规划

项目的范围就是对项目的界限进行定义。

范围规划就是确定项目范围，明确项目的主要可交付成果，制定项目范围管理计划，记载如何确定、核实与控制项目范围，以及如何制定和定义WDS。

4.2.1 项目范围规划的依据
  1. 环境因素
  2. 组织过程资产
  3. 项目章程
  4. 项目管理计划
4.2.2 项目范围管理计划

项目范围计划描述了对项目范围如何进行管理，项目范围怎样变更才能够与项目要求相一致..等问题，是项目管理团队确定、记载、核实、管理和控制项目范围的指南。

4.2.3 软件项目范围计划

包含4个方面

  1. 确定详细的项目范围
  2. 根据详细项目范围得到WBS
  3. 验收已经交付了的项目成果
  4. 控制并管理范围变更

4.3 收集需求和范围定义

4.3.1 需求与收集需求

【缺少正确的项目需求、定义和范围核实是导致项目失败的最主要原因】

让项目干系人积极参与需求发掘和分解需求的工作。

  1. 业务需求
  2. 干系人需求
  3. 解决方案需求
  4. 过度需求
  5. 项目需求
  6. 质量需求
4.3.2 收集需求的依据与成果

依据：范围管理计划、需求管理计划

成果：需求文件的形式

4.3.3 定义项目范围

根据范围管理计划，采取一定的方法，逐步得到精确的项目范围。

将项目主要的可交付成果细分为较小的便于管理的部分。

最主要成果：项目范围说明书

任何一个项目都有三个主要约束条件：质量、时间、成本。

相互影响制约，往往由于范围影响了时间和成本（如果一开始范围小，完成时间和成本就小）

一般来说，在项目初期应该提出一个比较稳定的项目范围，作为实施的框架也是后期项目管理的边界，但可能会存在项目边界模糊、扩大的现象，可能会导致项目管理和控制的难度变大，甚至失控。（根据约束关系模型C=f(Q,T,S)，C是成本，扩大项目范围的话 使得原有固定的S边界限制不固定，使约束出现复杂状况）

4.3.4 定义范围的依据
  1. 已有的文件（章程、需求文件等）
  2. 项目范围定义中收集的信息（及时更新的环境因素）
4.3.5 项目范围说明书

是对项目范围、主要可交付成果、假设条件和制约因素的描述

4.3.6 软件项目范围定义

软件系统的范围经常表现为软件需求规格说明书（SRS，Software Requirement Specifications）。

包含三个内容：

  1. 功能特征描述
  2. 系统接口描述
  3. 质量特征描述

4.4 工作分解结构技术WBS

4.4.1 用途

WBS确定了整个项目范围，将其有条理的进行组织，把项目工作分成了较小和更加容易管理的多项工作，每下降一个层次都意味着对项目更详尽的说明。

WBS主要依据：组织过程资产、项目范围说明书、项目范围管理计划、批准的变更请求。

4.4.2 制作WBS

模板法、分解法、自上而下、自下而上方法

模板法

​ 借用行业内标准化的项目分解结构模板，并根据实际情况进行必要的增删。

​ PMI工作分解结构实用标准是制作、深化和应用WBS的指南

分解法

将项目可交付成果分解成较小的便于管理的部分，直到工作细目（工作包）

工作包是WBS的最底层。

如果工作包的工作费用和持续时间是可估算出来的，就认为这种WBS是有效的，要在很远的将来完成的可交付成果或者子项目可能无法分解。

过细的分解可能造成管理精力的无效耗费，资源利用效率不高甚至降低实施该工作的效率。

4.5 项目范围核实与控制

范围核实：利益相关者对项目范围的正式接受

范围控制：对项目范围变更的控制

4.5.1 项目范围核实

核实项目范围包括或审查可交付成果，确保每一项结果都令人满意。

4.5.2 项目范围控制

项目范围变更的原因：

  1. 需求不明确
  2. 系统实施时间过长
  3. 用户业务需求改变
  4. 系统正常升级

需要考虑：

  1. 范围控制是必须的，不存在无变化的项目
  2. 项目范围变化，不仅仅意味着工作量增加
  3. 项目范围控制的目的不是阻止变更的发生。其主要任务是在提出变更请求后，管理相关计划、资源安排以及项目成果，使得项目各部分可以很好地配合在一起，消除变更带来的不利影响。
  4. 只有积极主动去进行项目范围控制，才能够控制、减少变更

措施

  1. 项目启动阶段的需求范围变更预防
  2. 项目实施阶段的项目范围变更（观念应该是 变更是必须的、可控的、有益的）
  3. 项目收尾阶段的总结（吸取教训）
4.5.3 软件项目范围变更控制

明确”必须做的“、”可以做的“、”不应该做的“

五、保障项目进度

5.3 定义活动

定义活动：识别和记录为完成项目可交付成果而需要采取的具体行动的过程

5.3.1 进一步分解项目工作

将工作包分解为活动，这些活动是能够保证完成交付产品项目的可实施的详细任务，而不是指可交付物。

列一个活动清单

5.3.2 项目活动的特征
  1. 对于可执行的活动，以动词/形容词+名词方式描述
  2. 如果一个资源分配给一项活动，应由一个人管理交付输出
  3. 每一项活动要定义好开始点
  4. 通常一项活动存在一个有形的输出或者完成的产品
  5. 活动在逻辑上与WBS元素相符
  6. 对于每一项活动要有重组的控制量和时间
  7. 开始和结束点必须要充分定义
  8. 能够计算实际成本
  9. 反应重要工作
  10. 零持续时间是里程碑胡总和事件，代表另一项活动或一组活动的开始或者完成

6.1 项目成本管理概述

6.1.1 项目成本与成本管理

项目成本：完成项目目标付出的费用或者消耗的资源。 是完成项目所需要的全部费用的总和。

软件项目成本分为开发成本和运行维护成本两大类。

成本管理：对成本进行规划、估算、预算、融资、筹资管理与控制的过程。

过程：规划成本管理、估算成本、制定预算、控制成本。

6.3 项目成本估算

6.3.1 成本估算的类型
  1. 量级估算：项目早期甚至在启动之前，精度-25%~75%
  2. 预算估算：项目完成前一到两年做出，精度-10~25%
  3. 最终估算：前一年或一年内，是最精确的，例如马上要采购某些东西，需要精确估算
6.3.2 成本估算的依据
